## 클라우드가 무엇인가?

**클라우드(Cloud)**, 정확히는 **클라우드 컴퓨팅(Cloud Computing)**은 인터넷을 통해 서버, 스토리지(저장 공간), 데이터베이스, 소프트웨어 등의 컴퓨팅 자원을 빌려 쓰는 기술을 말합니다.

쉽게 비유하자면 **'정수기 렌털 서비스'**와 비슷합니다. 직접 비싼 정수기를 사고 필터를 교체하며 관리할 필요 없이, 매달 사용료만 내면 업체에서 알아서 설치해주고 관리해 주는 것과 같은 이치입니다.

클라우드를 이해하기 위해 가장 중요한 특징과 장점은 다음과 같습니다.

### 1. 주요 장점

- **비용 절감 (Pay-as-you-go):** 값비싼 하드웨어나 서버 장비를 직접 구매하고 유지보수할 필요가 없습니다. 내가 사용한 만큼만 요금을 내면 됩니다.
    
- **뛰어난 접근성:** 인터넷만 연결되어 있다면 스마트폰, 태블릿, PC 등 어떤 기기에서든 언제 어디서나 내 데이터와 프로그램에 접근할 수 있습니다. (예: 구글 드라이브, 네이버 마이박스)
    
- **유연성과 확장성 (Scalability):** 갑자기 웹사이트에 방문자가 몰려 서버를 늘려야 할 때, 클릭 몇 번만으로 즉시 서버 용량을 늘릴 수 있고, 필요 없어지면 다시 줄일 수도 있습니다.
    
- **관리의 편의성:** 서버의 물리적인 고장 수리, 보안 업데이트, 백업 등을 클라우드 서비스 제공자(구글, 아마존, 마이크로소프트 등)가 알아서 처리해 줍니다.
    

### 2. 우리가 이미 쓰고 있는 클라우드의 예시

클라우드는 전문가들만 쓰는 어려운 기술이 아니라 이미 일상 깊숙이 들어와 있습니다.

- **스트리밍 서비스:** 넷플릭스, 유튜브, 멜론 등은 영상을 내 기기에 다운로드하지 않아도 클라우드 서버에서 실시간으로 불러와 재생합니다.
    
- **파일 저장:** 구글 사진, iCloud 등에 사진을 올리면 내 폰을 잃어버려도 새 폰에서 그대로 사진을 볼 수 있습니다.
    
- **업무 툴:** 구글 문서(Google Docs)나 노션(Notion)처럼 인터넷 브라우저만 켜면 바로 작업하고 자동 저장되는 서비스들도 모두 클라우드 기반입니다.
    

### 3. 클라우드 서비스의 3가지 형태

어떤 수준까지 빌려 쓸 것인가에 따라 크게 세 가지로 나뉩니다.

- **IaaS (인프라 빌려주기):** 가장 뼈대가 되는 컴퓨터(서버)와 저장 공간만 빌려줍니다. 운영체제부터 프로그램까지 사용자가 직접 설치해야 합니다. (예: AWS EC2, Google Compute Engine)
    
- **PaaS (플랫폼 빌려주기):** 개발자들이 앱이나 웹을 쉽게 만들 수 있도록 운영체제와 개발 환경까지 세팅해서 빌려줍니다.
    
- **SaaS (소프트웨어 빌려주기):** 완성된 소프트웨어를 인터넷을 통해 바로 사용하게 해줍니다. 일반 사용자들이 가장 많이 접하는 형태입니다. (예: 지메일, 마이크로소프트 365)
    

요약하자면, 클라우드는 **"내 방에 있는 컴퓨터의 한계를 벗어나, 전 세계에 연결된 거대한 슈퍼컴퓨터의 능력을 내 것처럼 빌려 쓰는 기술"**이라고 할 수 있습니다.


## 클라우드 환경에서 발생할 수 있는 취약점?

클라우드 환경은 기존의 전통적인 온프레미스(On-premise) 인프라와는 구조와 운영 방식이 다르기 때문에, 발생하는 취약점의 양상도 달라졌습니다. 특히 클라우드 네이티브 환경으로 넘어오면서 공격 표면(Attack Surface)이 훨씬 복잡하고 다양해졌습니다.

### 1. 잘못된 보안 설정 (Misconfiguration)

클라우드 보안 사고의 가장 압도적인 원인입니다. 클라우드 제공자(AWS, GCP 등)가 아무리 인프라를 튼튼하게 지켜도, 사용자가 설정을 실수하면 쉽게 돌파당합니다.

- **스토리지 퍼블릭 노출:** 클라우드 스토리지(예: AWS S3 버킷)나 데이터베이스의 접근 권한을 '전체 공개'로 잘못 설정하여 내부의 민감한 데이터가 그대로 유출되는 경우입니다.
    
- **과도한 권한 부여:** 특정 서비스나 사용자에게 필요 이상의 과도한 권한(IAM Role)을 부여하여, 계정 하나가 탈취되었을 때 인프라 전체로 피해가 확산되는 권한 상승(Privilege Escalation) 문제가 자주 발생합니다.
    

### 2. 컨테이너 및 오케스트레이션 취약점 (Cloud-Native Risks)

Docker와 같은 컨테이너 기술이나 이를 관리하는 Kubernetes 환경에서 발생하는 취약점입니다.

- **취약한 컨테이너 이미지:** 보안 패치가 안 된 구버전 라이브러리나, 심지어 악성코드가 은닉된 베이스 이미지를 검증 없이 그대로 가져와 컨테이너로 배포하는 경우입니다.
    
- **컨테이너 탈출 (Container Escape):** 웹 애플리케이션 등의 취약점을 통해 공격자가 컨테이너 내부로 침투한 뒤, 격리 환경을 뚫고 호스트 OS(노드)나 다른 컨테이너로 권한을 상승시켜 넘어가는 치명적인 공격입니다.
    
- **오케스트레이션 설정 오류:** Kubernetes API 서버가 인증 없이 외부에 노출되어 있거나 권한 통제가 미흡할 경우, 클러스터 전체의 제어권을 탈취당할 수 있습니다.
    

### 3. 소프트웨어 공급망 공격 (Supply Chain Attacks)

클라우드 생태계는 수많은 오픈소스, 써드파티(Third-party) 모듈, 외부 API들을 결합하여 구성되므로, 이 연결 고리인 공급망 자체가 강력한 타겟이 됩니다.

- **CI/CD 파이프라인 침해:** 빌드 및 배포를 자동화하는 파이프라인(Jenkins, GitHub Actions 등)의 권한을 탈취하여, 정상적인 배포 프로세스 중간에 악성 코드나 백도어를 주입하는 방식입니다.
    
- **오픈소스 종속성 취약점:** 서비스 개발에 사용된 외부 패키지나 라이브러리에 숨겨진 알려진 취약점(CVE)을 악용하여, 클라우드 환경 내부에 이상 행위를 유발하거나 데이터를 빼냅니다.
    

### 4. 안전하지 않은 인터페이스 및 API

클라우드 인프라의 생성, 관리, 제어는 모두 API를 통해 이루어집니다.

- API의 인증 및 인가(Authorization) 로직에 결함이 있거나(예: BOLA - Broken Object Level Authorization), 불필요한 내부 시스템 정보까지 과도하게 반환할 경우 주요 공격 통로가 됩니다. 웹 해킹 관점에서도 클라우드의 노출된 API를 대상으로 한 정찰 및 공격 시도가 매우 빈번합니다.
    

### 5. 하드코딩된 자격 증명 (Hardcoded Secrets)

- 애플리케이션 소스 코드나 컨테이너의 환경 변수(env) 파일 내부에 API 키, 데이터베이스 패스워드, 클라우드 접속 토큰 등을 하드코딩해 두었다가 GitHub 같은 퍼블릭 저장소에 유출되는 사고입니다. 클라우드에서는 키 하나만 탈취당해도 인프라 전체를 조작하거나 리소스를 무단으로 생성(크립토 재킹 등)할 수 있어 매우 위험합니다.
    

클라우드 보안의 핵심은 **'공동 책임 모델(Shared Responsibility Model)'**에 있습니다. 물리적인 서버나 네트워크 장비의 보안은 클라우드 제공자가 책임지지만, 그 위에 올라가는 애플리케이션의 코드, 컨테이너 이미지, 접근 권한 통제, 그리고 공급망에 대한 보안은 전적으로 사용자가 직접 관리하고 탐지해야 합니다.

## 클라우드 데이터셋을 구하기 어려운 이유?

클라우드 환경의 데이터셋, 특히 이상 탐지나 공격 분류를 위한 양질의 데이터를 구하는 것은 전 세계 보안 연구자들이 공통적으로 겪는 가장 큰 장벽 중 하나입니다.

단순히 데이터가 부족해서라기보다는, 클라우드라는 환경의 특수성 때문에 '연구에 쓸 수 있는 형태'로 만들기 어렵기 때문입니다. 그 핵심적인 이유들은 다음과 같습니다.

### 1. 기업의 민감 정보 및 규제 문제 (Privacy & Compliance)

실제 운영되는 클라우드 환경의 데이터에는 고객의 개인정보(PII), 기업의 핵심 비즈니스 로직, 인프라 아키텍처, 서비스 접근 키 등 치명적인 민감 정보가 포함되어 있습니다. 기업 입장에서는 연구 목적이라 하더라도 이를 외부에 공개하는 것은 막대한 법적, 비즈니스적 위험을 초래하기 때문에 실제 트래픽 로그를 공유하는 것을 극도로 꺼립니다.

### 2. 익명화(Anonymization)로 인한 유효 피처(Feature) 손실

기업이 데이터를 공개하더라도 보안을 위해 IP 주소, 페이로드(Payload) 내용, 사용자 식별자 등을 마스킹하거나 해시 처리합니다. 문제는 머신러닝이나 데이터 마이닝을 통해 이상 징후를 탐지할 때 이러한 정보들이 매우 중요한 **'피처(Feature)'**로 작용한다는 점입니다. 철저하게 익명화된 데이터는 컨텍스트가 훼손되어, 모델의 탐지 성능을 제대로 평가하기 어렵게 만듭니다.

### 3. 클라우드 네이티브 아키텍처의 복잡성과 휘발성

현재의 클라우드는 거대한 하나의 서버가 아니라, 수많은 마이크로서비스(MSA)와 컨테이너가 얽혀 있는 구조입니다.

- 컨테이너는 필요에 따라 생성되었다가 몇 분 만에 사라지기도 합니다.
    
- 네트워크 트래픽도 외부와의 통신(North-South)뿐만 아니라, 클러스터 내부의 컨테이너 간 통신(East-West)이 방대하게 일어납니다. 이처럼 동적이고 복잡한 환경에서 발생하는 모든 호스트 로그, 네트워크 패킷, API 호출 로그를 일관성 있게 수집하고 시간순으로 정렬하여 하나의 데이터셋으로 묶어내는 것 자체가 엄청난 엔지니어링 비용을 요구합니다.
    

### 4. '현실적이고 정교한' 공격 시나리오의 부재

정상 트래픽 로그는 어떻게든 수집할 수 있지만, 정작 모델을 학습시키기 위해 필요한 **'라벨링된 공격 데이터(Labeled Attack Data)'**를 구하기가 어렵습니다. 단순한 포트 스캐닝이나 디도스(DDoS) 같은 고전적인 공격 데이터는 만들기 쉽지만, 컨테이너 탈출, 클라우드 API 권한 상승, 혹은 복잡하게 얽힌 클라우드 공급망 공격 같은 은밀하고 다단계적인(Multi-stage) 위협은 인위적으로 시뮬레이션하고 라벨링하기가 매우 까다롭습니다. 결국 연구자들은 '현실 세계의 공격'과 '인위적으로 생성된 공격 데이터' 사이의 괴리를 고민할 수밖에 없습니다.

---

이러한 이유들 때문에 완벽한 실제 환경의 데이터셋을 구하기보다는, 특정 시나리오(예: 쿠버네티스 환경, 특정 클라우드 서비스)에 맞춰 공개된 벤치마크 데이터셋(예: ALIEN, CIC-Cloud-Security 등)을 활용하거나 도커 등을 이용해 통제된 소규모 환경을 구축하여 직접 데이터를 합성(Synthesis)하는 방식을 많이 취하게 됩니다.

## 쿠버네티스란?

단순하게 정의하자면 쿠버네티스는 **'수많은 컨테이너를 지휘하는 오케스트레이션(Orchestration) 도구'**입니다.

도커(Docker) 같은 기술로 애플리케이션을 '컨테이너'라는 상자에 예쁘게 포장하는 것까지는 좋았는데, 서비스 규모가 커지면서 이 컨테이너가 수십, 수백 개로 늘어나자 사람이 일일이 관리하기 불가능해졌습니다. "어느 서버에 새 컨테이너를 띄울까?", "컨테이너가 갑자기 죽으면 어떻게 다시 살리지?", "트래픽이 몰리면 어떻게 늘리지?" 같은 복잡한 문제들을 자동으로 해결해 주는 똑똑한 관리자가 바로 쿠버네티스입니다.

가장 핵심적인 3가지 특징은 다음과 같습니다.

1. **자동화된 복구 (Self-healing):** 쿠버네티스는 항상 사용자가 명령한 **'원하는 상태(Desired State)'**를 유지하려고 노력합니다. 만약 "웹 서버 파드(Pod) 3개를 유지해라"라고 설정해 두었는데 서버 하나가 다운되어 파드가 2개가 되면, 쿠버네티스가 이를 즉시 감지하고 다른 정상적인 노드(서버)에 새로운 파드를 1개 더 띄워 3개를 맞춥니다.
    
2. **오토 스케일링 (Auto Scaling):** 이벤트가 터져서 갑자기 접속자가 폭주하면, CPU나 메모리 사용량을 감지하여 자동으로 파드의 개수를 늘려 트래픽을 분산시킵니다. 트래픽이 줄어들면 다시 원래대로 줄여 클라우드 비용을 아껴줍니다.
    
3. **로드 밸런싱 (Load Balancing):** 파드가 여러 개 띄워져 있을 때, 외부에서 들어오는 요청을 어느 한 파드에 몰리지 않게 골고루 분산시켜 줍니다.

## kubectl 이란?

`kubectl`(쿠버네티스 컨트롤)은 쿠버네티스 클러스터와 통신하여 내부 리소스를 생성, 조회, 수정, 삭제할 수 있게 해주는 핵심 명령줄 도구(CLI)입니다.

클라우드 네이티브 환경에서 보안 분석을 하거나 데이터를 수집할 때 가장 기본이 되는 도구이기도 합니다. 목적에 따라 가장 많이 쓰이는 필수 명령어들을 정리해 드립니다.

### 1. 리소스 조회 및 상태 확인 (정보 수집)

가장 빈번하게 사용하는 명령어들로, 클러스터 내부에 어떤 자원들이 어떻게 돌아가고 있는지 파악할 때 사용합니다.

- **`kubectl get pods`**: 현재 네임스페이스에 있는 파드(Pod)들의 목록과 간략한 상태를 보여줍니다. (`-A` 옵션을 붙이면 모든 네임스페이스의 파드를 조회합니다.)
    
- **`kubectl get nodes`**: 클러스터를 구성하는 노드(서버)들의 상태를 확인합니다.
    
- **`kubectl get services`**: 실행 중인 네트워크 서비스 목록을 확인합니다.
    
- **`kubectl describe pod [파드_이름]`**: 특정 파드의 IP, 사용 중인 컨테이너 이미지, 할당된 노드, 그리고 **최근 발생한 이벤트(에러 로그 등)** 등 아주 상세한 정보를 출력합니다. 이상 현상 분석 시 필수적입니다.
    

### 2. 디버깅 및 로그 분석 (동적 분석 및 트러블슈팅)

컨테이너 내부의 상태를 확인하거나 직접 접근해야 할 때 사용합니다.

- **`kubectl logs [파드_이름]`**: 해당 파드 내에서 동작하는 애플리케이션의 표준 출력(stdout) 로그를 보여줍니다. 데이터셋을 수집하거나 비정상적인 행위를 탐지할 때 1차적으로 확인하는 곳입니다. (실시간으로 보려면 `-f` 옵션 사용)
    
- **`kubectl exec -it [파드_이름] -- /bin/bash`** (또는 `/bin/sh`): 동작 중인 파드 내부로 직접 쉘(Shell) 접속을 합니다. 컨테이너 내부의 파일 시스템을 뒤지거나 네트워크 상태를 점검하는 등 동적 분석 시 유용합니다.
    

### 3. 리소스 생성, 수정 및 삭제

클러스터의 상태를 변경하는 명령어입니다.

- **`kubectl apply -f [파일명.yaml]`**: YAML 파일에 정의된 명세대로 리소스(파드, 서비스, 권한 등)를 생성하거나 업데이트합니다. 선언적 관리를 위해 가장 권장되는 방식입니다.
    
- **`kubectl delete pod [파드_이름]`**: 특정 리소스를 삭제합니다. (YAML 파일을 지정하여 `kubectl delete -f [파일명.yaml]` 형태로도 많이 씁니다.)
    

### 4. 보안 및 권한 점검 (RBAC 확인)

클라우드 보안 관점에서 현재 계정이나 서비스 어카운트(Service Account)의 권한을 점검할 때 유용합니다.

- **`kubectl auth can-i [동사] [리소스]`**: (예: `kubectl auth can-i create pods`) 현재 내 자격 증명으로 특정 작업을 수행할 권한이 있는지 점검합니다. 과도한 권한 부여(Privilege Escalation) 취약점을 확인할 때 쓰입니다.

## secrets, pods란?

### 1. 파드 (Pod): 쿠버네티스의 가장 작은 '실행 단위'

도커(Docker)를 쓰실 때는 '컨테이너'를 하나씩 띄워서 실행하셨을 텐데, 쿠버네티스는 컨테이너를 직접 다루지 않고 **'파드'라는 껍데기(포장지)**로 한 번 더 감싸서 관리합니다.

- **비유하자면 '아파트'입니다:** 컨테이너가 아파트에 사는 '사람'이라면, 파드는 '아파트 호실'입니다. 보통 한 호실에 한 명(1 컨테이너)이 살지만, 필요에 따라 두세 명(여러 컨테이너)이 같이 살 수도 있습니다.
    
- 같은 파드 안에 있는 컨테이너들은 **하나의 IP 주소**를 공유하고, 저장 공간(볼륨)도 같이 나누어 씁니다. 우리가 띄우는 웹 서버, 데이터베이스 서버 등은 모두 이 '파드' 형태로 쿠버네티스 위에서 돌아가고 있습니다.
    

### 2. 시크릿 (Secret): 민감한 정보를 담아두는 '금고'

애플리케이션(파드)이 돌아가다 보면, DB 비밀번호, 클라우드 API 키, TLS/SSL 인증서 같은 **절대 유출되면 안 되는 민감한 정보**들이 필요합니다.

- 이런 정보들을 소스 코드나 도커 이미지 안에 하드코딩해 두면, 소스가 유출될 때 비밀번호도 같이 털리게 됩니다.
    
- 그래서 쿠버네티스는 이런 민감한 정보들만 따로 모아서 안전하게(?) 보관하는 전용 리소스를 만들었는데, 그것이 바로 **시크릿(Secret)**입니다.
    

### 3. 파드와 시크릿은 어떻게 만날까? (보안의 핵심)

파드(웹 서버)가 데이터베이스에 연결하려면 결국 시크릿(금고) 안에 있는 비밀번호를 알아야 합니다.

쿠버네티스는 파드를 실행하는 순간, **시크릿에 보관된 비밀번호를 꺼내서 파드 내부로 살짝 밀어 넣어줍니다.** 보통 파드 내부의 '환경 변수(Environment Variable)'로 세팅해 주거나, 파드 내부의 특정 폴더에 파일 형태로 마운트(Volume Mount)해 줍니다.
