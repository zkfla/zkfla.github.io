## 퍼징으로 찾지 못하는 취약점?
 
퍼징은 크래시·메모리 오류처럼 **명확한 오류 신호**가 있어야 탐지할 수 있습니다.
프로그램이 정상적으로 실행되지만 잘못된 동작을 하는 경우는 감지하지 못합니다.
 
---

## 우선 **퍼징(Fuzzing)**이란?

퍼징은 **자동화된 소프트웨어 테스팅 기법**입니다.
대상 프로그램에 예상치 못한 무작위·변형 입력을 대량으로 주입하여 크래시, 메모리 오류, 보안 취약점 등을 찾아내는 **행위(방법론)** 자체를 의미합니다.



---
 
## 1. 로직 기반 취약점 (Logic Vulnerabilities)
 
프로그램이 크래시 없이 실행되지만, 설계된 의도와 다르게 동작하는 취약점입니다.
퍼저 입장에서는 정상 실행처럼 보이기 때문에 탐지가 불가능합니다.
 
### 인증 우회 (Authentication Bypass)
 
잘못된 권한 검사 로직으로 인해 인증 없이 보호된 리소스에 접근 가능한 취약점입니다.
 
```
예시: 관리자 페이지에 접근할 때 토큰 검증 로직에 결함이 있어
      일반 사용자가 /admin 경로에 접근 가능한 경우
```
 
- 퍼저는 응답이 200 OK로 정상 반환되면 취약점으로 인식하지 못함
- 크래시가 발생하지 않기 때문에 오류 신호 자체가 없음

### 접근 제어 오류 (IDOR, Insecure Direct Object Reference)
 
다른 사용자의 데이터에 접근할 수 있는 구조적 결함입니다.
 
```
예시: /user/1234/profile 대신 /user/5678/profile 요청 시
      다른 사용자의 개인정보가 정상 응답(200 OK)으로 반환되는 경우
```
 
- 퍼저는 서버가 정상 응답을 돌려주면 "문제 없음"으로 판단
- 응답 내용이 올바른지 의미적으로 검증하는 Oracle이 없음

### 비즈니스 로직 결함 (Business Logic Flaws)
 
잘못된 계산, 잘못된 상태 전이 등 프로그램의 핵심 로직에 존재하는 결함입니다.
 
```
예시: 쇼핑몰에서 수량을 음수(-1)로 입력했을 때
      결제 금액이 환불되는 방향으로 계산되는 경우
```
 
- 프로그램은 정상적으로 실행되고 크래시도 발생하지 않음
- 결과가 비즈니스적으로 올바른지는 퍼저가 판단할 수 없음

---
 
## 2. 암호화·인증 관련 취약점 (Cryptographic Vulnerabilities)
 
암호화 구현의 의미적 오류는 프로그램이 정상 실행되기 때문에 퍼징으로 탐지할 수 없습니다.
 
### 취약한 암호 알고리즘 사용
 
| 취약한 알고리즘 | 문제점 |
|---|---|
| MD5, SHA-1 | 충돌 공격에 취약 |
| RC4 | 통계적 편향으로 키 복원 가능 |
| DES | 키 길이(56bit)가 너무 짧아 브루트포스 가능 |
 
- 알고리즘을 사용하는 코드 자체는 정상 실행됨
- 어떤 알고리즘을 쓰는지는 코드 리뷰나 정적 분석으로 확인해야 함

### 잘못된 난수 생성
 
```c
// 취약한 코드: 예측 가능한 난수 사용
srand(time(NULL));
int session_token = rand();  // 공격자가 시간 기반으로 예측 가능
```
 
- 코드는 정상 실행되고 크래시도 없음
- 난수의 예측 가능성은 퍼저가 판단할 수 없음

### 하드코딩된 비밀키·패스워드
 
```python
# 취약한 코드: 자격증명이 소스코드에 평문으로 존재
DB_PASSWORD = "admin1234"
SECRET_KEY = "hardcoded_secret_key"
```
 
- 실행 시 오류가 발생하지 않음
- 소스코드 분석이나 시크릿 스캐너(TruffleHog, GitLeaks 등)로 탐지해야 함

## 3. 레이스 컨디션 (Race Condition)
 
멀티스레드 환경에서 실행 순서에 따라 발생하는 취약점입니다.
 
```
스레드 A: 잔액 확인 (100원 있음)
스레드 B: 잔액 확인 (100원 있음)  ← 동시 실행
스레드 A: 100원 출금
스레드 B: 100원 출금              ← 합계 200원 출금됐지만 잔액은 0원
```
 
- 퍼징은 기본적으로 단일 입력에 집중하기 때문에 타이밍 의존 버그 재현이 어려움
- 발생 조건이 타이밍에 따라 달라지므로 일관된 재현이 어려움
- Thread Sanitizer(TSan)와 조합하면 일부 탐지 가능하지만 완전하지 않음

---
 
## 4. 시맨틱(Semantic) 취약점
 
프로그램이 크래시 없이 정상 실행되지만, 의미적으로 잘못된 동작을 하는 취약점입니다.
 
### SQL Injection
 
```sql
-- 입력: ' OR '1'='1
SELECT * FROM users WHERE id='' OR '1'='1'
-- 결과: 모든 사용자 데이터 반환 (크래시 없음)
```
 
- 쿼리가 정상적으로 실행되고 결과를 반환하기 때문에 퍼저는 오류로 인식하지 못함
- 별도의 Oracle(응답 내용 검증 로직)을 직접 설계해야 탐지 가능

### XSS (Cross-Site Scripting)
 
```html
<!-- 입력: <script>alert(1)</script> -->
<!-- 출력: 그대로 HTML에 삽입 → 브라우저에서 실행 -->
```
 
- 서버는 정상적으로 응답을 반환
- 브라우저에서 스크립트가 실행되는지는 퍼저가 알 수 없음

### 정보 노출 (Information Disclosure)
 
- 에러 메시지에 스택 트레이스, 내부 경로, DB 구조 등이 포함
- 서버는 정상 응답(200 OK 또는 500)을 반환하지만 내용에 민감 정보가 있음

---
 
## 5. 환경·설정 의존 취약점 (Configuration Vulnerabilities)
 
코드가 아닌 환경·설정에서 비롯되는 취약점은 퍼징의 탐지 범위 밖입니다.
 
| 취약점 | 예시 |
|---|---|
| 잘못된 서버 설정 | HTTPS 미적용, 불필요한 포트 오픈 |
| 의존성 취약점 | 알려진 CVE가 있는 오래된 라이브러리 사용 |
| 클라우드 권한 오류 | S3 버킷 퍼블릭 노출, IAM 과도한 권한 |
| 컨테이너 설정 오류 | root 권한으로 실행되는 Docker 컨테이너 |
 
---
 
## 6. 퍼징 한계 요약 및 보완 방법
 
| 취약점 유형 | 퍼징 탐지 | 보완 방법 |
|---|---|---|
| 버퍼 오버플로, UAF 등 메모리 오염 | ✅ 가능 | — |
| 포맷 스트링, 정수 오버플로 | ✅ 가능 (ASan·UBSan 연동 시) | — |
| 인증·인가 로직 결함 | ❌ 불가 | 수동 침투 테스트, 코드 리뷰 |
| 비즈니스 로직 오류 | ❌ 불가 | 수동 침투 테스트, 코드 리뷰 |
| 레이스 컨디션 | ❌ 어려움 | TSan, 동시성 테스트 도구 |
| 암호화 취약점 | ❌ 불가 | 정적 분석(SAST), 코드 리뷰 |
| SQL Injection / XSS | ❌ 어려움 | DAST 도구 (Burp Suite 등) |
| 하드코딩된 자격증명 | ❌ 불가 | 시크릿 스캐너 (GitLeaks, TruffleHog) |
| 설정·환경 오류 | ❌ 불가 | 보안 설정 감사, CSPM 도구 |
 
> **결론:** 퍼징은 메모리 안전성 버그 탐지에 강력하지만, 로직·의미론적 취약점에는 한계가 있습니다.
> 완전한 보안 테스트를 위해서는 코드 리뷰, 정적 분석(SAST), 동적 분석(DAST), 수동 침투 테스트를 함께 활용해야 합니다.
